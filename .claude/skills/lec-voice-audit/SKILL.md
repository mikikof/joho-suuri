# lec-voice-audit — 講義スライドの voice 自己審査

完成した HTML を読んで、 講義スライドとして自然か (AI 臭が出ていないか) を自動で点検し、 修正提案を出す。

---

## いつ使うか

- `new-lecture` skill の Phase 1 (HTML) 完成直後、 **ユーザー確認の前に必ず**
- `lec-review-slide` skill が新スライドを追加した直後
- ユーザーが「voice チェック」「AI 臭ないか見て」「lec-voice-audit」 と言ったとき

---

## 動作原則

1. **教材 HTML を Read のみ** (修正は出さない、 提案だけ)。 web search は禁止
2. **検出と修正提案を 2 段で出す**: 検出箇所一覧 → 各々の修正案
3. ユーザーが採否を決める。 Yes と言われたら Edit で適用 (または別 skill に委譲)
4. 検出結果が **0 件のときも明示**して「voice 観点では問題なし」 と報告

---

## 検出カテゴリ (Tier A / B / C)

### Tier A (必ず指摘・最優先)

#### A-1. 大主語 + 抽象動詞メタファー
**NG パターン**: 「世界」「現実」「真理」「未来」「人類」「歴史」 等の大主語 + 「書く」「動かす」「語る」「変える」「描く」 等の抽象動詞

例 (lec05 で実際に出た):
- ✗ 「3 種類のグラフで**世界を書いてきた**」
- ✓ 「身の回りの現象を、 3 種類のグラフで表してきた」
- ✗ 「式で**世界を動かす**」
- ✓ 「式で身の回りの現象を表す」

#### A-2. スローガン調 / 誇大表現
**NG パターン**: 「武器」「叩き台」「軸」「選ばれし」「最前線」「次の世代」「ベスト」「最強」「究極」 等の比喩・誇大語

例 (既存メモリより):
- ✗ 「君たちは選ばれしグループ」
- ✗ 「武器を授ける」「武器は授けた。次は使え」
- ✗ 「叩き台は、叩かれて完成する」
- ✗ 「ここで一気に空気を温める」

#### A-3. 体育会系・熱量過剰
**NG パターン**: 「一気に」「叩く」「刺す」「出し切れ」「笑ったら勝ち」 等の体育会系

#### A-4. 二重修飾 (読みにくい)
**NG パターン**: 「Aの BのC」 のように「の」 が連続して 2 つの修飾が「C」 にかかる構造

例 (lec05 で実際に出た):
- ✗ 「音は、 **空気の 1 秒間に 440 回の振動である**」 (「空気の」 と「1 秒間に 440 回の」 が両方 「振動」 を修飾)
- ✓ 「音は、 空気が 1 秒間に 440 回振動する現象である」 (動詞ベースに直す)

### Tier B (内容次第で指摘)

#### B-1. 対句アフォリズム
**NG パターン**: 「AはBより、Cする方が強い」「Xが変われば、Yも変わる」 のような短い対句が AI 臭の典型。 ただし 1〜2 回ならOK、 多用が NG。

#### B-2. 命令形・体言止め多用
**NG パターン**: 「動かしてみよう」「考えよう」 等の体育会調命令形、 体言止めの連発

許容される自然な体:
- 「皆さん」 + 中立動詞・宣言文 (「皆さんは… を確かめる」「ここでは… を扱う」)
- 「である」 調 (大学講義の standard)

#### B-3. 過剰カタカナ
**NG パターン**: 「メンバー」「メッセージ」「ベース」 等、 漢語+やまと言葉で表せる場面でカタカナに頼る

例 (既存メモリ daisu_column_natural_japanese より):
- ✗ 「メンバーが集まる」 → ✓ 「人が集まる」
- ✗ 「メッセージを伝える」 → ✓ 「考えを伝える」

### Tier C (注意・必要なら提案)

#### C-1. 「君ら」「諸君」 などの古風な呼びかけ
講義スライドでは「皆さん」 が標準。 これは強制ではないが、 統一すべき。

#### C-2. 抽象動詞メタファーの軽度版
「立ち上がる」「湧き上がる」 のような抽象動詞 (column 記事禁止 + 講義スライドでも避けたい)

#### C-3. 「(数値)で書ける」 と「○種類の波で書ける」 のような断定表現が大主語を伴う場合
「たった ○ つで書ける」 という強調が誇張になっていないか確認

## 実装手順

### Step 1: HTML を Read

対象ファイル (例: `lec05/index.html`) を Read。

### Step 2: 検出 (Bash の grep で抽出)

#### A-1 (大主語+抽象動詞) の検出
```bash
F="path/to/lec0N/index.html"
echo "=== A-1: 大主語+抽象動詞 ==="
grep -nE '(世界|現実|真理|未来|人類|歴史).*(書く|書ける|書いて|動か|語|描く|描いて)' "$F"
```

#### A-2 (スローガン調) の検出
```bash
echo "=== A-2: スローガン調 ==="
grep -nE '(武器|叩き台|選ばれし|最前線|次の世代|最強|究極|刺す|出し切)' "$F"
```

#### A-3 (体育会系) の検出
```bash
echo "=== A-3: 体育会系 ==="
grep -nE '(一気に|叩|刺す|出し切れ|笑ったら勝ち|空気を温め)' "$F"
```

#### A-4 (二重修飾) の検出
```bash
echo "=== A-4: 二重修飾 ==="
# 「Xの Yの Z」 のパターン (簡易版、 誤検出多めなので人間レビュー必須)
grep -nE 'の[^の。、<>]{1,15}の[^の。、<>]{1,15}の[^の。、<>]{1,15}(である|だ|です)' "$F"
# 数字を含む二重修飾の頻出パターン
grep -nE 'の [0-9]+ ?[^の。、<>]{1,10}の(振動|変化|動き|現象)' "$F"
```

#### B-2 (命令形多用) の検出
```bash
echo "=== B-2: 命令形 ==="
grep -ncE '(してみよう|考えよう|やってみよう|動かそう|見てみよう)' "$F"
# 件数を表示。 3 件以上で要検討
```

検出結果を一覧化。

### Step 3: 各検出箇所の修正提案

各検出について、 以下の形式で提案:

```
[Tier A-1] line 1475: 「3 種類のグラフで世界を書いてきた。」
  問題: 大主語 (世界) + 抽象動詞メタファー (書く)
  修正案: 「身の回りの現象を、 3 種類のグラフで表してきた。」
  根拠: 既存メモリ feedback_no_ai_tone_in_lectures (大主語+抽象動詞は禁止)
```

### Step 4: サマリ出力

```
=== voice 自己審査結果 ===
Tier A: N 件 (必ず修正推奨)
Tier B: M 件 (内容次第)
Tier C: K 件 (注意)

総合判定: {要修正 / 軽微 / 問題なし}
```

### Step 5: ユーザーに採否を確認

「これらを修正しますか? それとも一部だけ?」 と聞く。 修正が承認されたら Edit で適用。

## 既存メモリとの連携

このskillは以下のメモリに従う:
- `feedback_no_ai_tone_in_lectures.md` — 「武器」「叩き台」「軸」「選ばれし」 等の禁止 + 「世界を書く」 大主語+抽象動詞の禁止 (lec05 で追加)

- `feedback_column_abstract_verbs.md` — 「立ち上がる」「湧き上がる」 等の抽象動詞メタファー禁止
- `feedback_daisu_column_natural_japanese.md` — 「メンバー」「メッセージ」 等の英語直訳カタカナ回避

検出パターンが追加されたら、 まずメモリを更新してから本 skill を参照する。

## 注意

- **教材 HTML 以外を読まない**: web search は禁止 (既存メモリ feedback_audit_textbook_grounded に従う)
- **完成 HTML を即修正しない**: 検出 → 提案 → ユーザー承認 → 修正 のフローを守る
- **正解が複数あるときは複数案を出す**: 「修正案 A: ... / 修正案 B: ...」
- **0 件のときも明示**: 「voice 観点では問題なし」 と必ず報告 (silent fail を避ける)
