# lec-content-check — 講義スライドの内容正確性チェック

完成した HTML を読んで、 教材として正確か (数値ミス・概念混同・用語不統一がないか) を点検し、 修正提案を出す。

---

## いつ使うか

- `new-lecture` skill の Phase 1 (HTML) 完成直後、 **ユーザー確認の前に必ず**
- `lec-voice-audit` と並んで Phase 1 自己審査の柱
- ユーザーが「内容チェック」「数値合ってる?」「lec-content-check」 と言ったとき

---

## 動作原則

1. **教材 HTML と `_source/` の正本のみ Read**。 web search は禁止 (既存メモリ `feedback_audit_textbook_grounded`)
2. **検出と修正提案を 2 段で出す**: 検出箇所一覧 → 各々の修正案 (計算根拠付き)
3. **数値は必ず手で再計算する** (うろ覚えで判定しない)
4. ユーザーが採否を決める
5. 0 件のときも明示

---

## 検出カテゴリ

### C-1. 数値ミス (Tier A、 最優先)

#### C-1a. 算術ミス
- 例題の答え、 統計値、 物理量が手計算で合うか確認
- 例: tan(30°) = 0.577、 1/440 = 0.00227 秒 = 2.27 ms、 など

#### C-1b. データの整合
- スライドのテキスト値と、 JS 内のデータ配列の値が合うか
- 例: lec05 で「振幅 10.6℃」 とテキスト、 JS 内のデータから計算すると (26.9-5.4)/2 = 10.75 → 10.8℃ → 不一致

#### C-1c. 単位
- ℃, Hz, 秒, m, ms など単位が正しいか
- 例: 「2.27 ミリ秒」 ✓、 「2.27 秒」 ✗

### C-2. 概念の混同 (Tier A)

#### C-2a. 中心 vs 平均
- sin フィットの「中心 C」 と「データの算術平均」 を混同していないか
- 中心 C = (最大値 + 最小値) / 2 (中点)
- 算術平均 = データの和 / 件数
- 一般に C ≠ 算術平均。 lec05 では「中心(年間平均) = 16.1℃」 と書いてあったが、 これは中点 (15.75 ≠ 16.15)
- 修正案: 「中心(夏のピークと冬の底の中点) = 16.1℃」 or 算術平均で 15.8℃ に揃える

#### C-2b. パラメータ数の漏れ
- 「あらゆる X は A・B・C の 3 つで記述できる」 のような断定で、 実は 4 個 (中心 C + 振幅 A + 周期 T + 位相 φ) 必要なのに 3 個と書いていないか
- 例: lec05 SUMMARY card 02 「振幅 A・周期 T・位相 φ の 3 つで記述できる」 → 中心 C が漏れている、 修正案: 「中心 C・振幅 A・周期 T・位相 φ の 4 つで記述できる」

#### C-2c. 用語の意味のすり替え
- 「振動」 vs 「振動の現象」、 「周波数」 vs 「周期」、 「角速度」 vs 「角度」 など

### C-3. 用語統一 (Tier B)

#### C-3a. 同一概念の表記揺れ
- PART 4 で「重ね合わせ」、 SUMMARY で「足し合わせ」 — 同義だが揺れている → 統一を提案
- 「『ラ』 の音」 と「ラ音」 — 同一スライド内で混在は避ける (異なるスライド間ならOK)

#### C-3b. 出典の不整合
- 「気象庁 1991-2020 平年値」 と書いて、 JS 内のデータが実際と微妙に異なる場合
- 数値の出典がない場合 (どこのデータ?)

### C-4. 「観覧車のときと同じ」 のような語りの整合性 (Tier A)

#### C-4a. 「同じ手順」 が実は違う
- 例: lec05 slide-13「観覧車のときと同じ 3 つのスライダーを動かし」 — 観覧車は (周期 T, 振幅 A, 中心 C)、 気温フィットは (振幅 A, 中心 C, 位相 φ) で 3 個目が違う
- 修正案: 「観覧車のときに学んだ振幅・中心に、 新しく『位相』 を加えた 3 つのスライダーを動かす」 など

### C-5. グラフと数式の整合 (Tier A)

- 式表示と JS の predict 関数が一致するか
- グラフのスケール (Y 軸 min/max) と stat 表示が一致するか
- 例: 「振幅 50m、 中心 55m」 → 最下点 5m, 最高点 105m → 直径 100m と整合 ✓

### C-6. 物理的・現実的な妥当性 (Tier B)

#### C-6a. 現実離れした例
- 「直径 100m の観覧車を 1 周 8 秒」 → 実在の観覧車は 1 周 15-20 分。 教育用としては OK だが、 注釈があると良い
- 「演習用に高速化した架空の観覧車」 等のフォローがあると親切

#### C-6b. 例題の値
- 例: 1.05^30 ≈ 4.32、 1/440 ≈ 0.00227 など、 具体的に計算する

## 実装手順

### Step 1: HTML を Read

対象ファイル (例: `lec05/index.html`) を Read。

### Step 2: JS 内のデータ配列を抽出

```bash
F="path/to/lec0N/index.html"
echo "=== JS 内のデータ配列 ==="
grep -nE 'const (TOKYO|CAFE|DATA|RAW|MEASUREMENTS)' "$F"
# その箇所周辺を Read して、 データ値を取り出す
```

### Step 3: スライド上の数値を抽出

```bash
echo "=== 統計値・数値の表示 ==="
grep -nE '(stat-value.*[0-9]|<em>.*[0-9].*</em>|<strong>[0-9])' "$F" | head -50
```

### Step 4: 手計算で検算

データ配列から実際の値を計算し、 スライド表示と比較。

例: 東京気温 (TOKYO_TEMP) のデータから:
- 平均 = (5.4 + 6.1 + ... + 7.7) / 12 = 15.75 ≈ 15.8℃
- 最大 = 26.9 (8月)、 最小 = 5.4 (1月)
- 中点 = (26.9 + 5.4) / 2 = 16.15 ≈ 16.2 ↔ 表示「16.1℃」
- 振幅 = (26.9 - 5.4) / 2 = 10.75 ≈ 10.8 ↔ 表示「10.6」 → **不一致**

### Step 5: SUMMARY のパラメータ数チェック

```bash
echo "=== SUMMARY card-02 (周期現象の式構造) ==="
# SUMMARY セクション (data-title="まとめ" or "三つの統一原理") を抽出
```
SUMMARY の card 02 で「○つで記述できる」 が出てくる。 本文中のフィット式 (例: `C + A × sin(2π(t-φ)/T + π/2)`) と照らし合わせ、 パラメータ数 (C + A + T + φ = 4 個) が一致するか確認。

### Step 6: 用語統一チェック

```bash
echo "=== 用語の表記揺れ ==="
grep -ncE '重ね合わせ' "$F"
grep -ncE '足し合わせ' "$F"
# 同義語の出現数を比較、 多用される方に統一を提案
```

### Step 7: 報告書フォーマット

```

   修正案: 「10.6」 → 「10.8」

   表示: 「年間平均(中心) = 16.1℃」
   検証: 年間算術平均 = 189.0 / 12 = 15.75℃ (JS の `TEMP_MEAN` コメントにも「約 15.8」)
         中点 = (26.9 + 5.4) / 2 = 16.15℃
   問題: 「年間平均」 と「中心 (= max+min の中点)」 が混同されている

   修正案: 「中心 C・振幅 A・周期 T・位相 φ の 4 つで記述できる」

### Tier B
4. 用語揺れ: 「重ね合わせ」 N 件 / 「足し合わせ」 M 件
   提案: PART 4 用語 (重ね合わせ) に統一する
```

## 注意

- **web search 禁止**: 数値の正解は教材内 (JS データ配列) から計算するか、 教科書/公式データ (_source/) から取る
- **正解がわからない場合は「要確認」 として報告**: 「気象庁平年値の振幅は?」 のような不明値は、 ユーザーに「気象庁データ取り直しますか?」 と聞く
- **誤検出を恐れない**: 怪しい箇所は全部リストアップ、 ユーザーが判定
- **0 件のときも明示**: 「内容観点では問題なし」 と必ず報告
